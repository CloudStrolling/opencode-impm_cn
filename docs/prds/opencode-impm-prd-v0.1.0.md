# PRD文档

**项目名称：** 我是项目经理
**项目英文名：** opencode-impm
**版本号：** v0.1.0
**日期：** 2026-05-27

---

## 产品概述

"我是项目经理"是一款基于OpenCode平台的AI驱动软件工程全流程开发插件。它以PM（Project Manager）Agent作为主编排器，通过自动化调用多个专业subagent（BA/SA/TL/TE/FE/BE/DE/DBA/CS/WS/TW），遵循**需求分析 → PRD定义 → 架构设计 → 技术规格 → 任务分解 → TDD编码 → 文档生成**的完整工程化流程，实现从需求输入到代码交付的端到端自动化。

产品核心理念是 **"工程化替代Vibe Coding"** —— 通过结构化流程、细粒度任务分解、上下文隔离和TDD驱动，确保代码质量高、文档齐全、可维护性强，减少AI编程中的幻觉和代码Bug。

## 目标用户

| 用户类型 | 角色描述 | 核心场景 |
|---------|---------|---------|
| **独立开发者** | 个人项目开发，缺乏完整工程化流程支持 | 输入需求描述，自动完成从设计到编码的全流程 |
| **小团队负责人** | 管理3-10人开发团队，希望标准化开发流程 | 通过结构化文档链管理项目，确保团队协作质量 |
| **AI编程探索者** | 希望用AI辅助但厌恶Vibe Coding无规划的方式 | 使用系统化工程方法，获得高质量、可维护的代码产出 |
| **OpenCode平台用户** | 熟悉OpenCode工具链的开发者 | 在OpenCode环境中一键启动AI项目经理 |

## 用户故事

### US-001: /impm 命令启动全流程开发

- **作为** 用户
- **我想要** 通过 `/impm` 命令输入需求提示词或需求文档地址，一键启动AI项目经理全流程开发
- **以便** 无需手动管理多个步骤，自动进入工程化开发流程
- **验收标准：**
  - [ ] `/impm` 命令可被OpenCode平台识别并响应
  - [ ] `/impm` 支持直接输入提示词作为需求
  - [ ] `/impm` 支持接收需求文档路径作为参数
  - [ ] `/impm` 支持混合输入：提示词 + 需求文档地址
  - [ ] 命令启动后，自动创建Git新分支作为当前修改分支
  - [ ] 命令启动后，自动激活PM Agent开始编排流程
- **优先级：** 高
- **关联需求：** 核心流程 - 启动调用

### US-002: 项目信息初始化与维护

- **作为** SA (Software Architect) subagent
- **我想要** 扫描项目结构，生成或更新 `project.md` 文档（含项目信息、编码规范、Project Map）
- **以便** 为后续所有流程提供标准化的项目基础信息
- **验收标准：**
  - [ ] 自动检查并创建 `docs/` 及其子目录（`specs/`、`requires/`、`tasks/`、`prds/`）
  - [ ] 当 `project.md` 不存在时，根据模板 `PROJECT-TEMPLATE.md` 生成完整文档
  - [ ] `project.md` 包含三部分：Project Info、Coding Conventions、Project Map
  - [ ] 当 `project.md` 已存在时，仅更新 Project Map 部分
  - [ ] Project Map 覆盖所有源代码文件，每个文件列出所有函数/类，每项用一句话描述
  - [ ] Coding Conventions 根据编程语言自动生成行业规范，简洁明了
  - [ ] 信息未明确时向用户提问补充
- **优先级：** 高
- **关联需求：** 核心流程 - 生成或更新project.md文档

### US-003: 需求文档生成

- **作为** BA (Business Analyst) subagent
- **我想要** 读取用户提示词和引用的需求文档，合并整理成结构化的需求文档（Markdown格式）
- **以便** 为后续PRD生成提供完整、清晰的需求基线
- **验收标准：**
  - [ ] 读取本次提示词和提示词中引用的需求文档内容
  - [ ] 生成包含「项目背景、功能需求、非功能需求、约束条件、假设与依赖」的结构化文档
  - [ ] 自动确认或生成语义化版本号
  - [ ] 无指定版本号时，读取 `docs/requires/` 下最新文件版本号，修订号+1
  - [ ] `docs/requires/` 无文件时使用默认版本号 v0.0.1
  - [ ] 保存到 `docs/requires/{项目名称}-requirement-v{x.x.x}.md`
  - [ ] 所有内容使用简体中文
- **优先级：** 高
- **关联需求：** 核心流程 - 生成需求文档

### US-004: PRD文档生成

- **作为** BA (Business Analyst) subagent
- **我想要** 读取需求文档和 `project.md`，生成UserStory格式的PRD文档
- **以便** 将需求转化为可执行的产品定义，指导后续架构和开发
- **验收标准：**
  - [ ] 每个功能需求至少映射为一个UserStory
  - [ ] 每个UserStory有唯一编号（US-001, US-002...）
  - [ ] 每个UserStory包含：角色、功能、价值、验收标准、优先级
  - [ ] 验收标准具体、可测试，覆盖正常流程和边界情况
  - [ ] 包含优先级排序表格
  - [ ] 包含非功能性需求章节
  - [ ] 保存到 `docs/prds/{项目名称}-prd-v{x.x.x}.md`
  - [ ] 若需求细节不明确时向用户提问确认
- **优先级：** 高
- **关联需求：** 核心流程 - 生成PRD文档

### US-005: 架构文档管理

- **作为** SA (Software Architect) subagent
- **我想要** 根据PRD文档和 `project.md`，生成或更新架构文档 `ARCHITECTURE.md`
- **以便** 为技术实现提供系统架构设计和组件划分方案
- **验收标准：**
  - [ ] `ARCHITECTURE.md` 不存在时，基于PRD和 `project.md` 生成新文档
  - [ ] `ARCHITECTURE.md` 已存在时，判断PRD是否涉及架构变更
  - [ ] 涉及架构变更时更新架构文档，不涉及时跳过
  - [ ] 架构文档覆盖：系统组件、模块划分、数据流、技术选型
  - [ ] 需要时可查询网络文档以获取技术信息
  - [ ] 架构信息不明确时向用户提问确认
- **优先级：** 高
- **关联需求：** 核心流程 - 生成架构文件

### US-006: 技术规格说明生成

- **作为** TL (Tech Lead) subagent
- **我想要** 读取PRD、架构文档和 `project.md`，生成技术规格说明文档
- **以便** 为每个模块提供详细的技术实现方案和接口定义
- **验收标准：**
  - [ ] 基于PRD、架构文档和 `project.md` 生成技术规格
  - [ ] 覆盖接口设计、数据模型、核心算法、依赖关系
  - [ ] 每个UserStory对应的技术实现方案有清晰描述
  - [ ] 保存到 `docs/specs/{项目名称}-spec-v{x.x.x}.md`
  - [ ] 技术细节不明确时向用户提问确认
- **优先级：** 高
- **关联需求：** 核心流程 - 生成spec

### US-007: 任务清单生成与管理

- **作为** TL (Tech Lead) subagent
- **我想要** 根据PRD、架构文档和spec，生成可追踪的任务清单（.md和.json双格式）
- **以便** PM Agent按优先级和依赖关系编排编码执行
- **验收标准：**
  - [ ] 任务清单包含主任务和子任务，每个任务有唯一任务代码
  - [ ] 标注任务间的上下游依赖关系
  - [ ] 每个主任务标注所属UserStory关联关系
  - [ ] 每个主任务标注任务类型（前端/后端/通用）
  - [ ] 每个任务包含测试方法和验收标准
  - [ ] 每个主任务有状态字段：未完成/执行中/已完成
  - [ ] 同时生成 `.md`（可读）和 `.json`（可解析）两种格式
  - [ ] 保存到 `docs/tasks/{项目名称}-task-v{x.x.x}.md` 和 `.json`
  - [ ] 技术信息不足时查询本地代码或网络文档
- **优先级：** 高
- **关联需求：** 核心流程 - 生成任务文件

### US-008: TDD驱动编码执行

- **作为** PM Agent（编排TE/FE/BE/DE/CS/WS多个subagent）
- **我想要** 按照任务清单逐一执行编码任务，采用TDD方式（先写测试后写实现）
- **以便** 保证代码质量高、测试覆盖全面、减少Bug
- **验收标准：**
  - [ ] 启动CS（Code Searcher）subagent查询本地相关代码
  - [ ] 启动WS（Web Searcher）subagent查询三方包文档
  - [ ] 启动TE（Test Engineer）编写测试用例和测试代码
  - [ ] 根据任务类型选择FE/BE/DE subagent编写实现代码
  - [ ] 测试代码通过后再提交实现代码（红-绿-重构循环）
  - [ ] 测试不通过时，将失败信息加入上下文重新编码直至通过
  - [ ] 编码完成后启动TW（Technical Writer）添加详细代码注释
  - [ ] 注释语言使用 `project.md` 指定的本地语言（简体中文）
  - [ ] 编码完成后提交代码到Git并更新任务状态
- **优先级：** 高
- **关联需求：** 核心流程 - 执行代码变更

### US-009: 项目文档与代码注释生成

- **作为** TW (Technical Writer) subagent
- **我想要** 在编码完成后生成项目文档（README、部署手册、环境配置等）并完善代码注释
- **以便** 项目可交付、可部署、可维护
- **验收标准：**
  - [ ] 根据 `project.md` 的本地语言设置生成文档（简体中文）
  - [ ] 生成 README.md 项目介绍和使用说明
  - [ ] 生成编译与部署手册
  - [ ] 生成环境配置说明
  - [ ] 生成部署脚本（如适用）
  - [ ] 代码注释完整，使用JSDoc格式标注公共API
  - [ ] 所有代码注释使用简体中文
  - [ ] 文档完成后提交Git并合并到主分支
- **优先级：** 中
- **关联需求：** 核心流程 - 文档与注释编写

### US-010: Git分支与版本控制集成

- **作为** PM Agent
- **我想要** 在开发流程中自动管理Git操作（创建分支、提交代码、合并分支）
- **以便** 代码版本可追溯，每次开发工作隔离在独立分支中
- **验收标准：**
  - [ ] 每次 `/impm` 启动时自动创建新的Git分支
  - [ ] 分支名称与当前开发任务对应（可包含版本号或特征名）
  - [ ] 支持查看当前分支状态
  - [ ] 任务完成后自动提交代码
  - [ ] 全部流程完成后自动合并分支
  - [ ] Git操作失败时有明确的错误提示
- **优先级：** 中
- **关联需求：** 核心流程 - 启动调用、文档与注释编写

### US-011: 语义化版本号统一管理

- **作为** 系统
- **我想要** 在文档生成和项目版本管理中自动维护统一的语义化版本号
- **以便** 所有产出文档版本一致、可追溯
- **验收标准：**
  - [ ] 支持解析语义化版本号字符串（主版本.次版本.修订号）
  - [ ] 支持版本号比较（判断新旧）
  - [ ] 无指定版本号时自动递增修订号
  - [ ] 所有生成文档（requirement/PRD/架构/spec/任务）使用同一版本号
  - [ ] 版本号支持从文件名中提取
  - [ ] 版本号格式校验（拒绝非法版本号格式）
- **优先级：** 中
- **关联需求：** 产出物 - 版本号管理

### US-012: 细粒度任务与上下文隔离

- **作为** PM Agent
- **我想要** 为每个工程步骤和子任务启动独立的subagent，仅传递本次任务的必要上下文材料
- **以便** 保持上下文干净，减少AI编程幻觉和代码Bug
- **验收标准：**
  - [ ] 每个subagent启动时只读取本次步骤必要的输入文档
  - [ ] BA subagent只读取需求相关材料
  - [ ] SA subagent只读取PRD和project.md
  - [ ] TL subagent只读取PRD、架构文档
  - [ ] 编码subagent通过context-builder工具获取精简上下文
  - [ ] 步骤之间通过标准化交付物（文档）对接
  - [ ] 前序步骤的输出是后续步骤的唯一输入依赖
- **优先级：** 高
- **关联需求：** 核心流程与最佳实践 - 上下文隔离

## 非功能性需求

### 1. 性能要求
- Subagent启动响应时间不超过5秒
- 文档读取/写入操作响应时间不超过2秒
- 项目扫描（Project Map生成）覆盖100个文件以内不超过10秒
- Git操作（分支创建/提交/合并）响应时间不超过5秒

### 2. 可用性要求
- 所有文档和交互输出使用简体中文（专业术语可保留英文）
- /impm 命令参数格式清晰易用，支持Tab提示
- 每个步骤完成后向用户输出明确的进度信息
- 错误发生时输出可理解的错误信息和处理建议

### 3. 可靠性要求
- 文档生成完整率100%，不能出现截断或遗漏
- 版本号管理准确，不允许出现版本号冲突或回退
- Git操作确保数据安全，不丢失未提交代码
- 任务状态更新原子性，不会出现状态不一致

### 4. 可维护性要求
- 代码注释完整，公共API必须有JSDoc注释
- 编码规范统一（命名、文件组织、代码风格）
- 文档链完整可追溯：需求 → PRD → 架构 → Spec → 任务 → 代码
- 每个文档包含版本号，支持回溯历史版本

### 5. 可扩展性要求
- Subagent角色体系支持灵活扩展（新增角色不影响现有流程）
- Skill机制支持自定义扩展，新的技能可注册到PM Agent
- 工具（Tool）注册机制开放，可添加新的自动化工具

### 6. 兼容性要求
- 与OpenCode平台现有工具链兼容
- 支持Windows（win32）开发环境
- 支持TypeScript/JavaScript项目类型
- Git操作兼容主流Git版本

---

## 约束条件

1. 所有代码和文档使用 **简体中文** 为主要语言
2. 编码语言为 **TypeScript**（主）和 **JavaScript**（辅）
3. 运行环境为 **Node.js**，模块系统使用 ES Module
4. 必须使用 **TDD（测试驱动开发）** 方式编码
5. 每次 `/impm` 启动必须创建新的 **Git分支**
6. 所有文档必须保存到 `docs/` 目录下对应子目录中

## 假设与依赖

- 假设用户已安装 **OpenCode平台** 及运行环境
- 假设项目目录已初始化 **Git仓库**
- 假设用户具有基本的 **Node.js/TypeScript** 开发能力
- 依赖 **OpenCode** 的 Agent 和 Skill 机制作为运行基础设施
- 依赖 **Git** 作为版本控制工具
- 依赖网络搜索MCP工具用于查询三方包文档
