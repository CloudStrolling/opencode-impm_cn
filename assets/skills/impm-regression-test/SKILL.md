# impm-regression-test 技能

## 触发词
impm-regression-test

## 何时使用
- **触发时机**：impm-coding 编码阶段完成后，由 PM 触发。
- **触发条件**：所有 task 的单元测试已通过，代码已合并到待回归分支。
- **适用范围**：适用于所有涉及代码变更的迭代版本（含热修复）。

## 角色与职责

| 角色 | 职责 |
|------|------|
| PM（项目经理） | 触发回归测试流程，接收测试结果，决定是否继续后续流程 |
| TE（测试工程师） | 执行回归测试的全部技术工作，包括用例设计、脚本编写、执行与报告 |

## 执行步骤

### Step 1：环境准备与资料收集
**执行者**：TE
**输入**：
- 当前版本号（如 v1.2.3）
- `docs/project.md`（项目概况与当前进度）
- `docs/architecture.md`（系统架构与技术栈）
- `docs/prds/` 目录下本次迭代涉及的所有 PRD 文件
- `docs/tasks/` 目录下本次迭代的 task 清单
- 代码仓库地址及待回归分支

**输出**：回归测试范围确认文档（隐含）

**能力要求**：
- 能准确识别本次迭代涉及的所有功能模块和边界影响范围
- 能判断哪些历史功能可能受到本次变更影响，纳入回归范围

---

### Step 2：生成/更新回归测试用例文档
**执行者**：TE
**输出文件**：`docs/testcase/{PROJECT_NAME}-testcase-v{X.Y.Z}.md`

**能力要求**：
- **用例设计能力**：
  - 覆盖所有本次迭代的新增功能（基于 task 清单）
  - 覆盖所有可能受影响的已有功能（基于架构变更分析）
  - 按优先级标注：P0（阻塞级）、P1（核心功能）、P2（一般功能）、P3（边缘场景）
  - 每个用例必须包含：前置条件、测试步骤、测试数据、预期结果
- **用例复用能力**：
  - 优先复用历史版本的测试用例，仅对变更部分进行更新
  - 历史用例若不再适用，明确标记为「废弃」并说明原因
- **模板规范**：严格遵循 `TESTCASE-TEMPLATE.MD` 格式

---

### Step 3：编写自动化测试脚本
**执行者**：TE
**输出**：测试脚本代码文件

**能力要求**：
- **技术适配能力**：根据项目技术栈选择合适的测试框架
  - 前端项目：Jest / Vitest / Cypress / Playwright
  - 后端项目：JUnit / pytest / Go test / Jest
  - 全栈项目：端到端测试 + 单元测试 + 集成测试
- **目录规范**：遵循项目已有测试目录结构，如无则按语言惯例创建
  - 例如：`tests/`、`__tests__/`、`src/test/`、`e2e/` 等
- **脚本质量**：
  - 每个 P0/P1 用例必须对应自动化脚本
  - 脚本需包含清晰的注释说明对应的用例编号
  - 脚本需具备可重复执行性（不依赖手动前置操作）

---

### Step 4：执行测试并记录结果
**执行者**：TE
**输出**：更新后的 `docs/testcase/{PROJECT_NAME}-testcase-v{X.Y.Z}.md`

**能力要求**：
- **执行规范**：
  - 按优先级从高到低执行（P0 → P1 → P2 → P3）
  - 每个用例执行后，在用例文档中记录：
    - **测试脚本**：脚本文件路径或代码片段
    - **测试过程**：执行环境、实际输入、关键日志
    - **测试结论**：`通过` / `失败` / `阻塞` / `跳过`
  - 失败的用例需附带详细的错误信息、复现步骤、截图/日志
- **异常处理**：
  - 遇到阻塞级（P0）用例失败，立即停止后续测试
  - 遇到非阻塞级失败，记录后继续执行剩余用例

---

### Step 5：结果汇总与状态更新
**执行者**：TE
**输出**：回归测试报告 + 更新 `docs/project.md`

**能力要求**：
- **报告能力**：生成结构化的回归测试摘要，包含：
  - 测试范围概述
  - 用例执行统计（总数 / 通过 / 失败 / 跳过）
  - 失败用例清单及缺陷描述
  - 风险评估与建议
- **文档更新**：将 `docs/project.md` 的当前进度更新为 `impm-regression-test`

---

### Step 6：结果反馈与流程控制
**执行者**：PM（基于 TE 的测试结果）

| 测试结果 | 处理方式 |
|---------|---------|
| 全部通过 | TE 返回「测试成功」，PM 执行后续流程（如发布、部署） |
| 存在失败 | TE 返回「测试失败」及详细报告，**自动停止流程**。PM 需：1）指派开发修复缺陷；2）修复完成后重新触发 impm-regression-test |

---

## 质量标准（Checklist）

- [ ] 测试用例覆盖所有本次迭代的 task
- [ ] 测试用例覆盖所有架构文档中标注的变更点
- [ ] P0/P1 用例 100% 配备自动化脚本
- [ ] 所有 P0 用例执行通过
- [ ] 测试文档格式符合模板规范
- [ ] `docs/project.md` 进度已更新
- [ ] 失败用例附带可复现的错误信息

## 常见错误与规避

| 错误场景 | 规避方法 |
|---------|---------|
| 仅测试新增功能，遗漏关联功能回归 | 基于架构文档做变更影响分析，明确波及范围 |
| 用例与脚本不同步 | 用例文档中的「测试脚本」字段必须实时更新 |
| 测试环境配置导致假失败 | 记录执行环境信息，确保环境可复现 |
| 失败信息过于简略无法定位 | 必须包含：错误日志、输入数据、预期 vs 实际 |

## 完成后提示
执行完毕后，向用户提示以下选项：
1. **执行下一步** — 严格按照 `../../commands/impm.md` 中流程步骤的序号顺序，继续执行下一个步骤。
2. **执行后续所有步骤** — 严格按照 `../../commands/impm.md` 中流程步骤的序号顺序，自动执行从下一步开始的所有剩余步骤
