# impm-sds-create 技能

## 触发词

impm-sds-create

## 何时使用

- PRD文档生成完成后
- 架构文档生成完成后（如有）
- 需要将PRD和架构转化为具体技术规格时

## 执行步骤

### 1. 读取前序文档

使用impm_doc_reader读取：

- PRD文档
- project.md
- architecture.md（如有）

### 2. 读取技术规格文件模板

从本技能所在目录（与SKILL.md同级）读取 SDS-TEMPLATE.MD 文件作为技术规格文件的参考模板。
> **注意：** SDS-TEMPLATE.MD 是一个通用设计模板，涵盖了设计文档除了详细设计以外的常见模块。该模板仅供参考，**不是必须逐节照搬**。应根据当前项目的实际业务情况（项目类型、规模、部署方式、技术复杂度等）进行以下处理：
> - **裁剪**：去除不适用于当前项目的文档模块（例如，OpenCode插件项目通常不需要传统部署架构、灾备恢复、TLS/KMS等企业级安全架构）
> - **选择**：只保留对当前项目有价值的模块
> - **调整内容**：对保留的模块按实际设计填充内容，而非直接套用模板占位符。
> - **无需详细设计**：详细设计的内容会在task中体现，此处无需详细设计。

### 3. 生成sds

根据PRD和架构文档，生成技术规格说明。内容要求：
| 章节 | 内容要点 |
| ------------- | --------------------------- |
| **1. 技术方案概述** | 技术定位，架构风格、核心工作流，设计原则，prd对应 |
| **2. 模块概要设计** | 模块划分、模块间相互关系 |
| **3. 数据设计** | 数据库模式、数据字典、ER 图、数据流，分库分表和缓存 |
| **4. 接口设计** | 模块间接口、API定义规范（输入/输出/异常）、通信协议、认证授权，错误码 |
| **5. 安全设计** | 威胁模型，认证机制，授权机制，数据安全，审计日志 |
| **6. 非功能需求设计** | 性能指标，可用性，可扩展性，可靠性，可观测性 |
| **7. 部署与运维设计** | ci/cd流程和其他 |
| **8. 风险与缓解措施** | 各类风险说明 |
| **9. 附录** | 其他说明与术语，字典表这些 |

接口必须包含函数签名、参数类型、返回值、错误场景

### 4. 需求确认

如有不明确的技术细节，向用户提问直至清晰。
如有需要，可以通过搜索本地代码（用CS subagent）和网上查询（用WS subagent）收集相关信息，完善文档。

### 5. 写入文档

使用impm_doc_writer将sds写入：
`docs/sds/{PROJECT_NAME}-sds-v{X.Y.Z}.md`
然后将docs/project.md 的当前进度改为：impm-sds

### 6. 返回文档位置

返回生成的文档路径给PM agent。

## 交付物

- docs/sds/{PROJECT_NAME}-sds-v{X.Y.Z}.md

## 注意事项

- 这里的sds要求比较简略，后续和其他文件一起生成task。task保持详细。
- 接口定义要包含参数类型和返回值
- 错误处理要覆盖所有已知异常场景
- 所有内容使用简体中文

## 完成后提示

执行完毕后，向用户提示以下选项：
1. **执行下一步** — 按照 `/assets/commands/impm.md` 中的步骤顺序，继续执行下一个步骤
2. **执行后续所有步骤** — 按照 `/assets/commands/impm.md` 中的步骤顺序，自动执行从下一步开始的所有剩余步骤
