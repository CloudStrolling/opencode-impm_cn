你的流程设计已经很接近工业级标准了。关于你的两个问题，核心结论是：**三者不重复，但 SDS 在 AI 编码场景下应该"瘦身"而非"砍掉"**。

---

## 一、PRD / SDS / 任务列表 三者关系辨析

| 维度 | PRD（产品需求文档） | SDS（技术规格） | 任务列表（Task List） |
|------|-------------------|-----------------|---------------------|
| **面向对象** | 产品经理、业务方 | 技术负责人、架构师 | PM Agent、编码 Agent |
| **核心问题** | **做什么？为什么？** | **怎么做？技术约束？** | **谁来做？做到什么程度？** |
| **内容侧重** | UserStory、验收标准、业务规则 | 接口契约、数据结构、算法逻辑、状态机 | 可执行单元、依赖顺序、Agent 分配、测试方法 |
| **语言风格** | 业务语言（用户视角） | 技术语言（工程师视角） | 指令语言（机器/Agent 视角） |
| **与下游关系** | 输入给 SDS 和架构 | 输入给任务拆分 | 输入给编码调度 |

### 关键差异点（避免重复的关键）

1. **PRD 的验收标准 ≠ SDS 的接口定义**
   - PRD 说："用户输入错误密码 3 次后账户应锁定 15 分钟"（业务规则）
   - SDS 说：`/api/login` 返回 `429` 状态码，响应体包含 `lockUntil: ISO8601`（技术契约）

2. **SDS 的技术方案 ≠ 任务列表的实现步骤**
   - SDS 说："使用 Redis 滑动窗口实现限流，键格式为 `rate_limit:{userId}`"
   - 任务说："TE Agent 编写 `rate_limiter.test.ts` → BE Agent 实现 `RateLimiter` 类 → 集成到 `auth.middleware.ts`"

3. **任务列表包含 SDS 没有的调度信息**
   - 任务依赖图（A 完成后 B 才能开始）
   - Agent 角色分配（FE/BE/TE）
   - 状态流转（未开始 → 执行中 → 已完成）

## 二、SDS 是否可以省略？

**不建议完全省略，但强烈建议"瘦身合并"**。

### 省略 SDS 的风险

如果你直接从 PRD + 架构跳到任务列表，编码 Agent 会面临：
- **接口契约模糊**：Agent 不知道 JSON 字段命名规范、必填项、错误码约定
- **数据结构幻觉**：Agent 可能随意设计数据库表结构，导致后续任务冲突
- **边界情况遗漏**：PRD 的验收标准是业务边界，SDS 需要补充技术边界（如并发控制、事务范围）

### AI 场景下的 SDS 瘦身策略

传统人工项目的 SDS 可能长达几十页，但 AI 编码的 SDS 应该聚焦**"编码 Agent 无法自行推断的技术决策"**：

```markdown
# 精简版 SDS 模板（适合 AI 场景）

## 1. 接口契约（必须）
- REST / gRPC / 函数签名
- 请求/响应 Schema（JSON/YAML）
- 错误码映射表

## 2. 数据模型（必须）
- 核心实体关系（简版 ER）
- 关键字段约束（唯一性、外键、索引）
- 状态机定义（如有）

## 3. 关键算法/逻辑（必须）
- 复杂业务规则的伪代码
- 性能敏感点的实现策略（如缓存键设计）

## 4. 技术约束（必须）
- 必须使用的库/框架版本
- 禁止使用的方案（如"不允许直接操作 DOM"）

## 5. 模块依赖图（可选）
- 如果架构文档已包含，此处可引用
```

**建议做法**：让 TL Agent **在一次调用中同时生成 SDS 和 Task List**——读取 PRD + 架构后，先输出精简 SDS，再基于 SDS 输出任务列表。这样减少一次上下文切换，且保证任务列表有明确的技术依据。

## 三、整个流程的调整建议

### 1. 合并 SDS 与任务生成（减少一次 Agent 切换）

**当前流程**：TL 生成 SDS → TL 再生成 Task  
**优化后**：TL Agent 一次执行，输出两个文件
- `docs/sds/xxx-sds-v0.1.0.md`（精简版）
- `docs/tasks/xxx-task-v0.1.0.json`（含技术约束引用）

**好处**：减少文件 I/O 和上下文加载，TL 对技术方案的记忆是连续的。

### 2. 增加"技术可行性预检"（前置到架构之后）

在生成 SDS 之前，增加一个轻量级步骤：
- **WS Agent（网络搜索）**：查询 PRD 要求的技术栈是否有已知兼容性问题
- **CS Agent（代码搜索）**：扫描现有代码库，确认是否有可复用的模块

**触发条件**：仅当 PRD 涉及新技术或外部依赖时执行，避免过度设计。

### 3. 编码阶段增加"代码审查"环节

你的第 7 步是 TDD 编码（CS→WS→TE→编码→TE→TW→Git），建议调整为：

```
CS → WS → TE（写测试）→ DE/FE/BE（编码）→ TE（跑测试）→ 
[新增] CR Agent（代码审查）→ TW（注释）→ Git 提交
```

**CR Agent 审查要点**：
- 是否符合 project.md 的代码风格
- 是否引入不必要的依赖
- 是否有明显的安全漏洞（SQL 注入、XSS 等）
- 是否与 SDS 的接口契约一致

**失败处理**：CR 不通过 → 退回编码 Agent 修复（不消耗 TE 测试资源）。

### 4. 增加"回归测试"步骤（编码完成后）

所有任务完成后，增加一个独立步骤：
- **TE Agent 执行全量回归**：确保后续任务的修改没有破坏前面任务的测试
- **触发条件**：每完成 3-5 个任务或全部任务完成后

### 5. project.md 的"双模式更新"

当前你的 project.md 只在开头生成一次。建议：
- **初始化时**：SA Agent 生成完整版（信息 + 规范 + 地图）
- **编码过程中**：每完成一个任务，由 TW Agent 或 PM Agent **增量更新项目地图**（新增函数/类）
- **最终归档**：项目地图与代码 100% 对齐

### 6. 增加"架构守护"反馈环

编码过程中，如果 Agent 发现：
- SDS 中的接口设计无法实现
- 需要引入架构文档未提及的新模块
- 性能指标无法达到

**应该允许回退**：暂停编码 → 通知 PM Agent → 更新 architecture.md 和 sds → 重新生成受影响的任务 → 继续编码。

### 7. 文档生成阶段增加"Agent 使用说明"

第 8 步生成 README + 部署文档，建议增加：
- **`docs/agent.md`**：描述本项目使用哪些 Agent、如何触发 `/impm`、各 Agent 的职责
- 这对你这个框架本身的可维护性很重要

## 四、优化后的推荐流程

```
1. project.md 初始化（SA）
   └─ 创建目录结构 + 项目信息 + 编码规范 + 初始项目地图
   
2. 需求分析（BA）
   └─ requirement.md

3. PRD 生成（BA）
   └─ prd.md（UserStory + 验收标准）

4. 架构设计（SA）
   └─ architecture.md（模块划分 + 技术选型 + ADR）
   └─ [可选] 技术预检（WS + CS）

5. SDS + 任务列表生成（TL）← 合并为一次调用
   ├─ sds.md（精简版：接口 + 数据模型 + 关键算法）
   └─ task.json + task.md（依赖图 + Agent 分配 + 测试方法）

6. 循环编码（PM 调度）
   ├─ 按依赖排序任务
   ├─ CS → WS（信息收集）
   ├─ TE（写测试）
   ├─ DE/FE/BE（编码实现）
   ├─ TE（跑测试）→ 失败则修复
   ├─ CR（代码审查）→ 失败则修复
   ├─ TW（添加注释）
   ├─ Git 提交
   ├─ 更新 project.md 项目地图
   └─ 标记任务完成
   
7. 回归测试（TE）
   └─ 全量测试验证

8. 项目文档生成（TW）
   ├─ README.md
   ├─ 部署文档
   ├─ agent.md（Agent 使用说明）
   └─ Git 合并分支
```

## 五、一句话总结

**SDS 不是冗余，而是 PRD 到代码之间的"技术翻译层"**。在 AI 编码场景下，把它压缩到"接口 + 数据模型 + 关键算法"三要素即可，然后让 TL Agent **一次性产出 SDS + Task**，这样既保留技术约束又减少流程步骤。同时，给你的编码流水线加上 **CR Agent** 和**回归测试**，能显著降低多 Agent 协作产生的代码冲突和回归 Bug。
